<objective>
Implement household invite link generation, public invite page, and invite redemption so the second person can join a household.

Purpose: This is the core sharing mechanism -- without it, no second person can access the household. The owner generates a one-time link from settings, the invitee opens it, signs up or logs in, and is automatically placed into the household.

Output: Server actions for invite CRUD, a public /invite/[token] page, a /invite/[token]/accept route handler, and an updated settings page with invite link generation.
</objective>

    Export three server actions:

    a) `createInvite()`: 
       - Call getHouseholdId() to get {supabase, user, householdId}
       - Verify user is owner: query household_members for user_id + household_id where role='owner'. If not owner, return `{ error: 'Only the household owner can create invites.' }`

    b) `redeemInvite(token: string)`:
       - Validate token with inviteTokenSchema.safeParse({ token })
       - Create supabase client, get user (redirect to /login if no user)
       - Check user is NOT already in a household (query household_members for user_id). If already in a household, return `{ error: 'You are already in a household.' }`
       - Query household_invites where token matches, redeemed_at IS NULL, expires_at > now(). If not found, return `{ error: 'This invite link is invalid or has expired.' }`
       - In a sequence: (1) Update household_invites SET redeemed_by=user.id, redeemed_at=now(). (2) Insert into household_members: household_id from the invite, user_id, role='member', **display_name: `user.email?.split('@')[0] ?? null`** (CRITICAL: this must be included so SHAR-04 "You and [partner name]" works — Plan 04-02 depends on display_name being populated for the joining member)
       - The household_members INSERT will fail RLS because only owners can insert members. To fix this, we need a migration (see step 3).
       - revalidatePath('/app/dashboard')
       - Return `{ data: { household_id: invite.household_id } }` on success

    c) `getActiveInvites()`:
       - Call getHouseholdId()
       - Query household_invites where household_id matches, redeemed_at IS NULL, expires_at > now(), order by created_at desc, limit 5
       - Return `{ data: rows }`

    **3. Create hearth-budget/supabase/migrations/20260422000001_invite_rls_fix.sql:**
    The current RLS policy "owners manage members" on household_members only allows INSERT when is_household_owner(household_id) is true. But the invitee is NOT the owner. We need an additional policy that allows a user to insert themselves as a member when they hold a valid, unredeemed invite token.

    Add a new RLS policy:
    ```sql
    -- Allow invited users to add themselves as members
    create policy "invited users join household"
    on household_members for insert
    to authenticated
    with check (
      user_id = auth.uid()
      and role = 'member'
      and exists (
        select 1 from household_invites
        where household_invites.household_id = household_members.household_id
          and household_invites.redeemed_by = auth.uid()
          and household_invites.redeemed_at is not null
          and household_invites.expires_at > now()
      )
    );
    ```

    IMPORTANT: The redeemInvite action must UPDATE the invite first (setting redeemed_by and redeemed_at), THEN insert the household_members row. The RLS policy checks that the invite has already been marked as redeemed by this user.

    Also add a policy so members can read their own membership:
    This is already covered by "members read membership" policy. No change needed there.
